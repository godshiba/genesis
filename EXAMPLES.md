# EXAMPLES.md — What GENESIS Actually Generates

A worked example so you can judge the output before installing. The project:
a small CLI pomodoro timer in TypeScript. The user ran `/genesis:init` in an
empty directory and answered the interview.

## The generated CLAUDE.md (abridged)

```markdown
# CLAUDE.md — pomo

A zero-config CLI pomodoro timer that survives terminal restarts.

Generated by /genesis:init on 2026-06-12 (GENESIS plugin).

## Stack
TypeScript 5 strict, Node 22, commander for args, no other runtime deps.
Vitest for tests. tsup for bundling.

## Commands
npm run dev          # run locally
npm run check        # THE validation gate - tsc + vitest + eslint (G2)
npm test             # tests only

## Architecture
cli (commander parsing)
 └── core/           pure: timer state machine, no I/O, no clocks
      └── persist/   impure shell: state file read/write
**Invariants (do not violate):**
1. core/ imports nothing from cli/ or persist/ - pure functions only.
2. All clock access goes through core/clock.ts - tests inject fake time.
3. State has one file (~/.pomo/state.json) and one writer (persist/store.ts).

## Conduct — how to work on any task
1. Surface assumptions before coding ... (the four rules, see template)

## The Seven Gates — a task is done only when all pass
1. G1 Registration ... 7. G7 Session (full table, see template)

## Out of Scope
- No GUI or menu-bar app - CLI only.
- No sync, accounts, or cloud anything.
- No plugin system for "custom timer strategies".
```

## The seeded registries

`docs/registry/DECISIONS.md` got the architecture choices, with losers:

```markdown
## 2026-06-12 — State machine as pure reducer

**Decision:** Timer logic is a pure (state, event) -> state reducer in core/.
**Context:** Timer must survive process restarts; logic must be testable
without waiting real minutes.
**Alternatives rejected:** setInterval-driven class (untestable without
mocking timers globally); SQLite persistence (a JSON file is enough for one
record - YAGNI).
**Consequences:** All side effects live at the edges; adding "pause" is a new
event type, not new plumbing.
```

`docs/registry/ISSUES.md` got the riskiest unknown from the investigation
step:

```markdown
### ISS-001 — Terminal notification reliability across platforms
**Severity:** deferred
**Context:** node-notifier behavior differs on macOS/Linux/WSL; unverified.
**Investigation path:** spike scripts/notify-test.ts on each target platform
before Phase 2.
```

`docs/registry/ROADMAP.md` got phases with gates; `SESSION_LOG.md` got the
genesis entry; `FILES.md` got a row for every scaffold file.

## What the hooks do during work

```text
[you create src/core/timer.ts without registering it]
  -> G1: 'src/core/timer.ts' has no row in docs/registry/FILES.md.

[Claude writes  // TODO: handle zero-length sessions]
  -> G3: new TODO without an ISS- reference - register it (/genesis:issue).

[Claude edits vitest.config.ts]
  -> G2: quality-gate config - fixing code beats weakening the gate.

[Claude runs git commit]
  -> G2: check command must be green for this commit.

[turn ends with changed code, session log untouched]
  -> G7 (blocks): append the handoff entry - /genesis:close does this.

[context window fills, Claude Code is about to compact the conversation]
  -> pre-compaction hook: writes docs/registry/.session-snapshot.md
     (branch, HEAD, every uncommitted file) before detail is summarized away.
```

## When a session is about to be cut off

Two cutoffs end a session; GENESIS keeps both resumable.

**Context fills (automatic).** Just before Claude Code compacts a long
conversation, the pre-compaction hook writes a breadcrumb to
`docs/registry/.session-snapshot.md`:

```markdown
# Pre-compaction snapshot (mechanical, auto-written by GENESIS)

Written 2026-06-13T11:42:08Z just before a auto context compaction.
This is a breadcrumb, not a handoff. If you see it on resume, the last
session may have ended mid-task before /genesis:close ran.

- Branch: feat/persist
- HEAD: a1b2c3d Phase 1 complete
- Uncommitted changes:
    M  src/persist/store.ts
    ?? src/persist/store.test.ts
```

If the session survives, you keep working. If it dies, the resume hook surfaces
this next session so the working tree is never silently lost.

**Usage cap hit (you trigger it).** The 5-hour and weekly limits are not exposed
to hooks, so there is no auto-trigger — you watch `/usage`. When the cap nears,
run `/genesis:close`; its emergency mode captures where mid-task work sits and
stops, without spending the last of your budget settling the tree:

```text
/genesis:close   (with /usage near the cap)
  -> records: src/persist/store.ts half-written, atomic-write path TODO,
     next = finish writeAtomic() then wire store.test.ts (ROADMAP p2 t1)
  -> appends SESSION_LOG handoff, stops. Resumes cleanly after the reset.
```

## What the next session sees

The session-resume hook injects this automatically at startup — before you
type anything:

```text
GENESIS resume context (auto-loaded from docs/registry/):

Last session handoff:
## 2026-06-12 — Phase 1 complete
- Reducer + clock injection landed in src/core/. npm run check green @ a1b2c3d.
- Next: persist/store.ts per ROADMAP phase 2 task 1.

Next roadmap task: implement persist/store.ts with atomic writes
Open issues: 1 (ISS-001)
```

No re-reading the tree. No "where were we?". That is the whole point.

# GENESIS

Операционная система проектов для Claude Code. Одна доктрина: **чего нет в
реестре — не существует.**

- **Постоянная память проекта** — шесть реестров хранят то, что теряет память
  диалога: решения, мины, проблемы, сессии, файлы, дорожную карту
- **Семь гейтов с реальными зубами** — хуки блокируют завершение сессии без
  записи передачи и подсвечивают нарушения в момент, когда они происходят
- **Ноль холодных стартов** — следующая сессия автоматически подгружает
  последнюю передачу и продолжает ровно с того места, где вы остановились

Выжато из шести реальных доведённых до релиза проектов — полная
доказательная база в [ANALYSIS.md](./plugins/genesis/ANALYSIS.md). Реальный
сгенерированный результат — в [EXAMPLES.md](./EXAMPLES.md).

## Установка

```text
/plugin marketplace add godshiba/genesis
/plugin install genesis@godshiba
```

Затем в любом проекте (новом или существующем):

```text
/genesis:init
```

## Какую проблему это решает

Каждая сессия Claude Code начинается с амнезии. Модель, которая знала, почему
вы выбрали SQLite, какой API врёт в своей документации и каким был следующий
шаг, — исчезает в момент завершения сессии. Многосессионные проекты платят
повторяющийся налог: перечитывание дерева, повторный вывод решений, наступание
на те же мины. А проекты, начатые без подготовки, платят второй налог:
непроверенные допущения о рамках, архитектуре и том, что значит «готово».

GENESIS устраняет оба. Знания проекта живут в файлах, которые переживают любую
сессию; гейты механически определяют «готово»; хуки принудительно исполняют
ключевые правила — дисциплина больше не зависит ни от чьей памяти, включая
память Claude.

## Как это работает

### 1. Подготовка до кода — `/genesis:init`

В новом проекте он берёт у вас интервью: цель одним предложением, жёсткие
ограничения, критерии успеха и — обязательно — минимум три явных «не делаем»,
потому что явный список не-целей — самая дешёвая защита от расползания рамок.
Затем он исследует самое рискованное неизвестное, проектирует архитектуру как
контракт зависимостей (какие импорты разрешены — всё остальное запрещено) и
разбивает сборку на фазы с приёмочными гейтами.

В существующем репозитории он читает вместо того, чтобы спрашивать: стек из
манифестов, команды из скриптов, историю из git — и встраивается вокруг того,
что уже есть. Существующий CLAUDE.md никогда не перезаписывается без
одобрения.

Результат: индивидуальный `CLAUDE.md` (до 150 строк) с инвариантами проекта,
правилами поведения, гейтами — и постоянная память проекта:

### 2. Реестры — `docs/registry/`

| Файл | На какой вопрос отвечает |
|------|--------------------------|
| `FILES.md` | Где что лежит? |
| `ISSUES.md` | Что сломано, отложено или ограничено? (`ISS-NNN`) |
| `DECISIONS.md` | Почему построено именно так? (только добавление, с отвергнутыми альтернативами) |
| `LANDMINES.md` | Обо что споткнётся следующая сессия? |
| `SESSION_LOG.md` | Что произошло и каков следующий конкретный шаг? |
| `ROADMAP.md` | В какой мы фазе и каков её гейт? |

Память диалога умирает вместе с сессией. Реестры — нет.

### 3. Ежедневный цикл

Фиксация стоит пять секунд — поэтому она реально происходит:

| Команда | Когда |
|---------|-------|
| `/genesis:landmine` | Что-то только что стоило вам десяти минут недоумения |
| `/genesis:decision` | Вы выбрали между реальными альтернативами |
| `/genesis:issue` | Баг или отложенная задача не должны жить в TODO-комментарии |
| `/genesis:gate-check` | Перед коммитом — аудит всех семи гейтов |
| `/genesis:close` | Завершение сессии — запись передачи |
| `/genesis:learn` | После релиза — вернуть уроки в систему |

### 4. Семь гейтов

Задача готова, только когда проходят все:

1. **Регистрация** — у каждого нового файла есть строка в `FILES.md`
2. **Валидация** — команда проверки проекта зелёная; на красном не продолжаем
3. **Проблема** — проблемы становятся записями `ISS-NNN`, а не TODO-комментариями
4. **Синхронизация документов** — изменил X — обнови документ Y в том же коммите
5. **Мина** — грабли записываются в момент, когда они стоили времени
6. **Решение** — неочевидный выбор записывается вместе с отвергнутыми вариантами
7. **Сессия** — работа завершается передачей: что изменилось, состояние репозитория, следующий шаг

Каждый гейт получает самое сильное исполнение, какое допускает его природа.
Хуки берут на себя механическое: G7 **блокирует** завершение хода, изменившего
код без записи в журнал сессий; G1 и G3 подсвечивают незарегистрированные
файлы и осиротевшие TODO; обязательство G2 внедряется перед каждым git commit,
а при правке конфигов тестов или линтера срабатывает страж конфигурации
(чинить код лучше, чем ослаблять гейт). Проектно-зависимые проверки (запуск
вашей команды проверки, аудит карты синхронизации документов) живут в
`gate-check` и `close`. Суждения (G5, G6) прощупывает агент `gate-auditor`,
помечая вероятные пропуски, а не притворяясь, что grep умеет распознать
решение.

Сгенерированные проекты также несут четыре **правила поведения** (по
наблюдениям Карпатого о типичных ошибках LLM): проговаривай допущения,
простота прежде всего, хирургические правки, шаги с проверяемой целью —
связанные с гейтами, а не наклеенные как лозунги.

### 5. Возобновление — замкнутый цикл

`/genesis:close` записывает передачу; **хук возобновления автоматически читает
её обратно** при старте следующей сессии — последняя передача, следующая
задача дорожной карты, открытые проблемы — всё внедрено в контекст до того,
как вы напечатаете первое слово. Как это выглядит — в
[EXAMPLES.md](./EXAMPLES.md).

## Для кого это

**Новичкам в Claude Code:** установите, запустите `/genesis:init`, ответьте на
вопросы. Вы получите проект, который объясняет сам себя, и сможете закрыть
ноутбук посреди фичи, не потеряв ничего.
[MASTER.md](./plugins/genesis/MASTER.md) — заодно короткий курс о том, зачем
существует каждая из этих практик.

**Профессионалам:** ценность — в механическом исполнении и нулевом дрейфе:
гейты как хуки, а не как пожелания; архитектура как контракт зависимостей;
явные критерии делегирования для мультиагентной работы; и цикл обратной связи,
превращающий опыт проектов в улучшения шаблонов. Всё — чистый markdown и
shell: форкните и сделайте своим.

## Совместимость

GENESIS — оболочка поверх существующей архитектуры Claude Code, а не замена
какой-либо её части. Каждый хук «падает открытым»: в проекте без
`docs/registry/` GENESIS полностью молчит. Он сосуществует с вашими другими
плагинами, агентами и навыками (включая обучающие системы вроде ECC — GENESIS
сохраняет в файлы репозитория, они — в собственное состояние; пересечений
нет). Хуки используют `git` и `python3`, когда те есть, и молчат, когда их
нет. По замыслу не привязан ни к какому стеку.

## Что в коробке

```text
plugins/genesis/
├── MASTER.md            доктрина — зачем существует каждая часть
├── ANALYSIS.md          вскрытие шести проектов, из которых выжат GENESIS
├── templates/           CLAUDE.template.md + шесть скелетов реестров
├── skills/              init, gate-check, close, landmine, decision, issue, learn
├── agents/              gate-auditor, genesis-architect
└── hooks/               страж сессии G7 (блокирующий) + загрузчик возобновления,
                         подсказки G1/G3, напоминание G2 + страж конфигурации
EXAMPLES.md              реальный сгенерированный результат от начала до конца
CHANGELOG.md             история версий
```

# GENESIS

Claude Code 的项目操作系统。唯一信条：**未登记的事物不存在。**

- **永久的项目记忆** — 六个登记簿保存对话记忆会丢失的一切：决策、地雷、问题、会话、文件、路线图
- **七道动真格的门禁** — 钩子会拦住没写交接记录就想收工的会话，违规当场提醒，而不是事后追悔
- **零冷启动** — 下一个会话自动加载上次的交接记录，从你停下的地方继续

提炼自六个真实上线的项目 — 完整的证据记录见
[ANALYSIS.md](./plugins/genesis/ANALYSIS.md)。真实生成结果见
[EXAMPLES.md](./EXAMPLES.md)。

## 安装

请逐条执行（这是两条独立的命令 — 粘贴成一行会导致 URL 解析失败）：

```text
/plugin marketplace add godshiba/genesis
```

```text
/plugin install genesis@godshiba
```

然后在任意项目中（新项目或已有仓库）：

```text
/genesis:init
```

## 它解决什么问题

每个 Claude Code 会话都从失忆开始。那个知道你为什么选 SQLite、知道哪个 API
的文档在撒谎、知道下一步是什么的模型 — 会话一结束就消失了。多会话项目因此
反复缴税：重读代码树、重新推导决策、重新踩同样的地雷。而没有前期准备就开工
的项目还要缴第二种税：关于范围、架构和"完成"标准的未经检验的假设。

GENESIS 同时消除这两种税。项目知识存放在比任何会话都长寿的文件里，门禁以
机械方式定义"完成"，钩子强制执行关键规则 — 纪律不再依赖任何人的记忆，
包括 Claude 的。

## 工作原理

### 1. 写代码之前先准备 — `/genesis:init`

新项目：它会采访你 — 一句话目的、硬性约束、成功标准，以及（强制的）至少三条
明确的"不做什么"，因为显式的范围排除清单是成本最低的防漂移手段。然后它调查
风险最大的未知项，把架构设计成依赖契约（哪些 import 被允许，其余一律禁止），
并把构建划分为带验收门禁的阶段。

已有仓库：它读取而不是提问 — 从 manifest 读技术栈、从脚本读命令、从 git
读历史 — 在现有基础上改造。它绝不会未经批准覆盖已有的 CLAUDE.md。如果仓库里
已经堆了一摊零散的文档，`/genesis:docs` 会来收拾：逐一清点并分类每个文档，
把决策记录和踩坑笔记吸收进登记簿，把文件移动、合并成清晰的结构，对照代码
标记过时的内容，并依据代码补写缺失的文档 — 一切先以迁移计划的形式呈给你
审批，通过后以一个可回滚的提交执行。

输出：一份量身定制的 `CLAUDE.md`（150 行以内），含项目不变量、行为准则、
门禁，以及项目的永久记忆：

### 2. 登记簿 — `docs/registry/`

| 文件 | 回答的问题 |
|------|-----------|
| `FILES.md` | 每样东西在哪里？ |
| `ISSUES.md` | 什么坏了、被推迟了、有局限？（`ISS-NNN`） |
| `DECISIONS.md` | 为什么这样构建？（只增不删，含被否决的备选方案） |
| `LANDMINES.md` | 什么会绊倒下一个会话？ |
| `SESSION_LOG.md` | 发生了什么？下一步具体做什么？ |
| `ROADMAP.md` | 现在处于哪个阶段？该阶段的门禁是什么？ |

对话记忆随会话死亡，登记簿不会。

### 3. 日常循环

记录只要五秒钟，所以你真的会去记：

| 命令 | 时机 |
|------|------|
| `/genesis:landmine` | 刚刚被一个坑白白耗掉十分钟 |
| `/genesis:decision` | 你在真实的备选方案之间做了选择 |
| `/genesis:issue` | 一个 bug 或推迟项不该活在 TODO 注释里 |
| `/genesis:status` | 任何时候 — 一屏看清现状：阶段、问题、上次交接 |
| `/genesis:gate-check` | 提交前 — 审计全部七道门禁 |
| `/genesis:close` | 结束会话 — 写交接记录 |
| `/genesis:learn` | 上线后 — 把经验沉淀回系统 |

### 4. 七道门禁

只有全部通过，任务才算完成：

1. **登记** — 每个新文件在 `FILES.md` 有一行记录
2. **验证** — 项目检查命令必须是绿的；红灯绝不继续
3. **问题** — 问题变成 `ISS-NNN` 条目，而不是 TODO 注释
4. **文档同步** — 改了 X 就要在同一次提交里更新文档 Y
5. **地雷** — 坑在让你付出代价的那一刻就被记录
6. **决策** — 非显而易见的选择连同被否决的方案一起记录
7. **会话** — 工作以交接记录收尾：改了什么、仓库状态、下一步

每道门禁都获得其性质所允许的最强执行方式。钩子负责机械可查的部分：G7 会
**阻止**一个改了代码却没碰会话日志的回合结束；G1 和 G3 在出现未登记文件或
孤儿 TODO 时即时提醒；G2 的义务在每次 git commit 前被注入，且编辑测试或
lint 配置时会触发配置卫兵（修代码胜过放水门禁）。项目相关的检查（运行你的
检查命令、审计文档同步映射）由 `gate-check` 和 `close` 负责。判断类的
（G5、G6）由 `gate-auditor` 代理主动探查，标记"疑似遗漏"而不是假装 grep
能检测出一个决策。

生成的项目还带有四条**行为准则**（源自 Karpathy 的 LLM 编码陷阱指南）：
先亮出假设、简单优先、外科手术式改动、目标驱动步骤 — 与门禁打通，
而不是贴标语。

### 5. 无缝续接 — 闭环

`/genesis:close` 写下交接；**会话恢复钩子在下一个会话启动时自动读回** —
上次交接、下一个路线图任务、未关闭的问题，在你打第一个字之前就已注入。
实际效果见 [EXAMPLES.md](./EXAMPLES.md)。

## 适合谁

**Claude Code 新手：** 安装、运行 `/genesis:init`、回答问题。你会得到一个
能自我解释的项目，可以在功能写到一半时合上电脑而什么都不丢。
[MASTER.md](./plugins/genesis/MASTER.md) 同时是一门关于这些实践为何存在的
速成课。

**专业用户：** 价值在于机械化执行和零漂移 — 门禁是钩子而不是口号、
依赖图即契约的架构产出、多代理协作的明确委派准则，以及把项目经验转化为
模板改进的反馈循环。全部是纯 markdown 和 shell — fork 它，改成你自己的。

## 兼容性

GENESIS 是 Claude Code 现有架构之上的外壳，不替代其中任何部分。所有钩子
失败时静默放行：在没有 `docs/registry/` 的项目里，GENESIS 完全沉默。它与
你的其他插件、代理和技能共存（包括 ECC 这类学习系统 — GENESIS 持久化到
仓库文件，它们持久化到自己的状态，互不重叠）。钩子在有 `git` 和 `python3`
时工作，没有时保持沉默。设计上与技术栈无关。

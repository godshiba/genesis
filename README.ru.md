# GENESIS

Операционная система проектов для Claude Code. Вся доктрина — в одной фразе:
**чего нет в реестре, того не существует.**

- **Память проекта, которая не обнуляется** — шесть реестров хранят всё, что
  забывает диалог: решения, грабли, баги, сессии, файлы, дорожную карту
- **Семь гейтов не для галочки** — хуки не дадут завершить сессию без записи
  о передаче дел и одёргивают в момент нарушения, а не постфактум
- **Никаких холодных стартов** — новая сессия сама подхватывает последнюю
  передачу и продолжает ровно с того места, где вы остановились

GENESIS вырос из шести живых, доведённых до релиза проектов — вся
доказательная база собрана в [ANALYSIS.md](./plugins/genesis/ANALYSIS.md).
Как выглядит результат на практике — в [EXAMPLES.md](./EXAMPLES.md).

## Установка

Выполняйте по одной за раз (это две отдельные команды — вставленные одной
строкой, они ломают URL):

```text
/plugin marketplace add godshiba/genesis
```

```text
/plugin install genesis@godshiba
```

Дальше — в любом проекте, хоть новом, хоть давно живущем:

```text
/genesis:init
```

## Зачем это нужно

Каждая сессия Claude Code начинается с чистого листа. Модель, которая
помнила, почему выбрали SQLite, какой API врёт в собственной документации и
что делать дальше, исчезает вместе с сессией. На длинных проектах этот налог
платишь снова и снова: перечитываешь дерево, заново выводишь старые решения,
наступаешь на те же грабли. А проект, начатый без подготовки, платит и второй
налог — непроверенные допущения о скоупе, архитектуре и том, что вообще
считать «готово».

GENESIS закрывает обе дыры. Знания проекта лежат в файлах, которые переживут
любую сессию; «готово» определяется гейтами, а не ощущениями; ключевые правила
исполняют хуки — дисциплина перестаёт держаться на чьей-то памяти, включая
память Claude.

## Как это устроено

### 1. Сначала подготовка, потом код — `/genesis:init`

В новом проекте он вас расспросит: цель одним предложением, жёсткие
ограничения, критерии успеха и — обязательно — минимум три пункта «этого не
делаем». Явный список не-целей — самый дешёвый способ удержать скоуп. Дальше
он разберёт самое рискованное неизвестное, опишет архитектуру как контракт
зависимостей (разрешены только перечисленные импорты, всё остальное —
нарушение) и нарежет работу на фазы с приёмочными гейтами.

В существующем репозитории он не спрашивает, а читает: стек — из манифестов,
команды — из скриптов, историю — из git, и аккуратно встраивается в то, что
уже есть. Ваш CLAUDE.md без одобрения никто не перепишет. А если в репозитории
накопилась разрозненная документация, `/genesis:docs` наведёт порядок: каждый
док инвентаризируется и классифицируется, записи о решениях и заметки о
граблях впитываются в реестры, файлы переносятся и сливаются в понятную
структуру, устаревшие утверждения сверяются с кодом, недостающие доки пишутся
по живому коду — всё это сначала показывается вам как план миграции, а после
одобрения выполняется одним откатываемым коммитом.

На выходе — `CLAUDE.md` под конкретный проект (до 150 строк): инварианты,
правила поведения, гейты — и постоянная память проекта:

### 2. Реестры — `docs/registry/`

| Файл | На какой вопрос отвечает |
|------|--------------------------|
| `FILES.md` | Где что лежит? |
| `ISSUES.md` | Что сломано, отложено, не работает? (`ISS-NNN`) |
| `DECISIONS.md` | Почему сделано именно так? (только дописывается, вместе с отвергнутыми вариантами) |
| `LANDMINES.md` | Где грабли, на которые наступит следующая сессия? |
| `SESSION_LOG.md` | Что сделано и какой следующий шаг? |
| `ROADMAP.md` | Какая сейчас фаза и что нужно для её закрытия? |

Память диалога умирает вместе с сессией. Реестры — нет.

### 3. Ежедневный цикл

Запись занимает пять секунд — поэтому она действительно делается:

| Команда | Когда |
|---------|-------|
| `/genesis:landmine` | Только что потеряли десять минут на ровном месте |
| `/genesis:decision` | Выбрали один вариант из нескольких настоящих |
| `/genesis:issue` | Багу не место в TODO-комментарии |
| `/genesis:status` | В любой момент — сводка на один экран: фаза, баги, последняя передача |
| `/genesis:gate-check` | Перед коммитом — прогон всех семи гейтов |
| `/genesis:close` | Конец сессии — записать передачу дел |
| `/genesis:learn` | После релиза — вернуть выводы в систему |

### 4. Семь гейтов

Задача закрыта, только когда проходят все семь:

1. **Регистрация** — у каждого нового файла есть строка в `FILES.md`
2. **Валидация** — команда проверки зелёная; с красной дальше не идём
3. **Баги** — проблемы оформляются как `ISS-NNN`, а не зарываются в TODO
4. **Синхронизация доков** — поменял X — в том же коммите обнови документ Y
5. **Грабли** — записываются в ту же минуту, когда о них споткнулись
6. **Решения** — неочевидный выбор фиксируется вместе с тем, что отвергли и почему
7. **Сессия** — работа заканчивается передачей дел: что изменилось, состояние репозитория, следующий шаг

Каждый гейт исполняется настолько жёстко, насколько позволяет его природа.
Механику закрывают хуки: G7 **не даст** завершить ход, в котором менялся код,
а журнал сессий остался нетронутым; G1 и G3 одёргивают за незарегистрированный
файл и бесхозный TODO; G2 напоминает о себе перед каждым `git commit`, а при
правке конфигов тестов или линтера срабатывает отдельный страж — чинить код
правильнее, чем ослаблять проверку. Всё проектно-зависимое (запуск вашей
команды проверки, сверка карты синхронизации доков) делают `gate-check` и
`close`. А там, где нужно суждение (G5, G6), работает агент `gate-auditor`:
он помечает вероятные пропуски, не делая вид, будто grep способен распознать
принятое решение.

В сгенерированный CLAUDE.md также входят четыре **правила поведения** (по
мотивам наблюдений Андрея Карпати о типичных провалах LLM): сначала проговори
допущения, проще — лучше, правки только хирургические, у каждого шага есть
проверка. Они привязаны к гейтам, а не висят лозунгами.

### 5. Возобновление — замкнутый круг

`/genesis:close` записывает передачу дел; **хук возобновления сам читает её
обратно** при старте следующей сессии: последняя передача, следующая задача из
дорожной карты, открытые баги — всё уже в контексте до того, как вы напечатали
первое слово. Наглядно — в [EXAMPLES.md](./EXAMPLES.md).

## Кому это нужно

**Если вы только начинаете с Claude Code:** поставьте, запустите
`/genesis:init`, ответьте на вопросы. Получите проект, который объясняет себя
сам, — можно захлопнуть ноутбук посреди фичи и ничего не потерять.
[MASTER.md](./plugins/genesis/MASTER.md) заодно работает как короткий курс:
почему каждая из этих практик вообще существует.

**Если вы давно в теме:** главное здесь — механическое исполнение и
отсутствие дрейфа. Гейты — это хуки, а не благие пожелания; архитектура
отдаётся контрактом зависимостей; для мультиагентной работы есть внятные
критерии делегирования; а цикл обратной связи превращает опыт проектов в
правки шаблонов. Внутри только markdown и shell — форкайте и переделывайте
под себя.

## Совместимость

GENESIS — надстройка над штатной архитектурой Claude Code, он ничего в ней не
подменяет. Любой хук при сбое молча пропускает дальше: в проекте без
`docs/registry/` GENESIS вообще никак себя не проявляет. Спокойно уживается с
другими плагинами, агентами и навыками (в том числе с обучающимися системами
вроде ECC: GENESIS пишет в файлы репозитория, они — в собственное состояние,
пересечений нет). Хукам нужны `git` и `python3` — если их нет, хуки просто
молчат. К стеку не привязан в принципе.

## Что внутри

```text
plugins/genesis/
├── MASTER.md            доктрина — зачем существует каждая часть
├── ANALYSIS.md          разбор шести проектов, из которых вырос GENESIS
├── templates/           CLAUDE.template.md + шесть скелетов реестров
├── skills/              init, docs, status, gate-check, close, landmine,
│                        decision, issue, learn
├── agents/              gate-auditor, genesis-architect, doc-curator
└── hooks/               страж сессии G7 (блокирующий) + загрузчик возобновления
                         + снимок перед сжатием контекста, подсказки G1/G3,
                         напоминание G2 + страж конфигов
tests/run.sh             тесты хуков (30 сценариев, гоняются в CI)
EXAMPLES.md              реальный результат генерации, от и до
CHANGELOG.md             история версий
```
